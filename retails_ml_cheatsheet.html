01

Metrics — pick the right one or you're done

If he asks "is this model good?" your first move is always to ask for the baseline and class distribution. Never quote accuracy alone.

The decision tree

ProblemUseWhy
Balanced classificationAccuracy + F1Both classes matter equally.
Imbalanced classificationPR-AUC, per-class F1ROC-AUC lies on imbalance — it stays high even with garbage minority recall.
FN expensive (cancer, fraud)Recall first, then PrecisionMissing positives is the costly error.
FP expensive (spam → inbox)Precision firstFalse alarms erode trust.
Ranking / retrievalMRR, NDCG, Recall@kPosition matters; binary metrics ignore order.
Regression — outliers matterRMSE, MAPERMSE penalizes large errors quadratically.
Regression — robustMAE, Median AEMAE treats all errors linearly; median ignores outliers.
Probabilistic outputLog loss, Brier scoreReward calibrated probabilities, not just decisions.
LLM generativeLLM-judge + human spot-check + task-specificNo single metric; layered eval is the only way.

Precision, recall, F1 — be able to derive these in your head

Confusion matrix         Predicted +   Predicted −
Actual +      TP           FN
Actual −      FP           TN
Formulas Precision = TP / (TP + FP)   — "of what I called positive, how much was right?"
Recall = TP / (TP + FN)   — "of all real positives, how many did I catch?"
F1 = 2 · (P · R) / (P + R)   — harmonic mean, punishes the weaker of the two
Accuracy = (TP + TN) / N   — useless on imbalanced data

The "ROC-AUC vs PR-AUC" trap — he WILL ask this

ROC-AUC plots TPR vs FPR. On heavy imbalance (say 1% positive), FPR stays tiny no matter what you do, so AUC stays optically high — even when the model never finds positives.

PR-AUC plots Precision vs Recall. Both involve TP, so it can't hide behind the majority class. On imbalanced problems, always report PR-AUC.

"On imbalanced data ROC-AUC is misleading because FPR is dominated by the majority class — I default to PR-AUC and per-class F1."
FLIP
Q: Your model gets 99% accuracy on fraud detection. Ship it?
A: No. Fraud is ~0.5% of transactions, so a model predicting "never fraud" gets 99.5%. Ask for PR-AUC, recall on the positive class, and the cost of a missed fraud vs. a false flag. Accuracy alone is meaningless under imbalance.

FLIP
Q: When would you choose F2 over F1?
A: F-beta with beta=2 weights recall more than precision. Use it when missing a positive is much more expensive than a false alarm — medical screening, fraud, security. F0.5 weights precision more — use when false alarms are expensive.

Statistics essentials — the stuff he assumes you know

An ML lead won't quiz you on textbook stats, but if you fumble z-scores or what a p-value means, the room cools. Lock these.

Mean, median, std — and which to use when

Formulas Mean μ = (1/n) Σ xᵢ
Variance σ² = (1/n) Σ (xᵢ − μ)²
Std deviation σ = √σ²   — same units as the data
Median = middle value when sorted   — robust to outliers
  • Mean + std — symmetric, roughly normal data.
  • Median + IQR — skewed data, outliers present, financial / latency / income.
  • If mean ≠ median, the distribution is skewed — investigate before modeling.

Z-score — the abnormal-value workhorse

Formula z = (x − μ) / σ

Tells you how many standard deviations a point is from the mean. Under a normal distribution:

|z|% of data insideInterpretation
≤ 1~68%Normal, within one std.
≤ 2~95%Common boundary for "normal range."
≤ 3~99.7%The classic 3-sigma rule; |z| > 3 = outlier candidate.
> 3< 0.3%Investigate — anomaly or data error.
"For unimodal roughly-normal data I use a 3-sigma rule on z-scores; for skewed data I switch to IQR or modified z-score with the median."

IQR — the robust alternative when data isn't normal

Formula IQR = Q3 − Q1
Outlier if x < Q1 − 1.5·IQR   or   x > Q3 + 1.5·IQR
Extreme outlier: same with 3·IQR

Used in box plots. Doesn't assume normality. Use this for income, latency, prices — anything skewed.

Modified z-score (median-based, robust)

Formula z_mod = 0.6745 · (x − median) / MAD
where MAD = median(|xᵢ − median|)

Use when the data has outliers that would distort the regular mean and std. Threshold: |z_mod| > 3.5.

Correlation — and the trap

Pearson r r = Σ((xᵢ − x̄)(yᵢ − ȳ)) / √(Σ(xᵢ − x̄)² · Σ(yᵢ − ȳ)²)
Range: −1 to +1. Linear relationships only.
  • Pearson — linear, requires roughly normal data.
  • Spearman — rank-based, captures monotonic relationships, robust.
  • Correlation ≠ causation. Two variables can correlate via a confounder (lurking variable) or pure coincidence.
  • Correlation between two features doesn't mean redundant — check VIF for multicollinearity in linear models.

p-value & significance — explain it correctly

p-value = probability of observing data this extreme if the null hypothesis were true.

It is NOT: the probability the null is true. It is NOT: 1 − probability the alternative is true. Get this wrong out loud and you lose technical credibility.

  • p < 0.05 — reject null (conventional, not magical).
  • Always pair with effect size — a tiny effect can be statistically significant with enough data and still be useless.
  • A/B tests: need pre-registered metric, sample size calc, no peeking, multiple-testing correction (Bonferroni, BH-FDR).

Confidence intervals — the practical version

A 95% CI for the mean: x̄ ± 1.96 · σ/√n. Means: if we repeated the sampling many times, ~95% of intervals would cover the true mean. Not "95% probability the true mean is in this interval" — that's Bayesian language.

Distributions you must recognize on sight

ShapeDistributionWhere you see it
Bell curveNormal / GaussianErrors, heights, measurement noise.
Long right tailLog-normal / ParetoIncome, prices, file sizes, session durations.
CountsPoissonEvents per interval (clicks, leads/hour).
0/1 eventsBernoulli / BinomialConversion, click-through.
Heavy-tailedPower lawWord frequency, network connections.

If your feature is heavy-tailed (income, revenue, durations), log-transform it before linear models. Tree models don't care.

FLIP
Q: Quick — define a z-score and when you wouldn't use it.
A: z = (x − μ) / σ. Number of standard deviations from the mean. Don't use it when data is heavily skewed or has outliers — μ and σ are themselves distorted. Switch to IQR or modified z-score using the median and MAD.

Outliers & abnormal values — detect, decide, defend

Outliers aren't always errors. The mature answer is: detect, then decide whether they're noise, mistakes, or signal.

Detection methods — pick by data shape

MethodAssumesUse when
Z-score (3σ rule)~Normal, unimodalQuick check on roughly Gaussian features.
Modified z-score (MAD)None — robustOutliers present that would skew μ/σ.
IQR · 1.5NoneSkewed data; the box-plot default.
Isolation ForestMultivariateMulti-dim outliers; tree-based, no distance metric needed.
DBSCANDensity-basedPoints in low-density regions = outliers.
One-class SVMNovelty detectionTrained on "normal," flag deviations.
Autoencoder reconstructionNN-basedHigh-dim (images, sequences); high reconstruction error = anomaly.
Domain rulesNoneBusiness knowledge — age > 120, negative price, etc.

The decision framework — what to DO with an outlier

  1. Investigate first, never auto-remove. Is it a data entry error, a sensor glitch, or genuine signal?
  2. Data error → fix or drop, document why.
  3. Genuine but rare → keep. In fraud, outliers ARE the signal. Removing them removes the problem.
  4. Distorts a linear model → transform (log, Box-Cox, Yeo-Johnson), winsorize (cap at percentiles), or use a robust model (gradient boosting handles this naturally).
  5. Document the choice. Reproducibility > cleverness.
"I never auto-drop outliers. I detect them with IQR or isolation forest, investigate whether they're errors or signal, and either fix the data or use a robust model — boosting handles outliers far better than linear regression."

Treatment options at a glance

Reduce influence
  • Winsorize — clip at 1st/99th percentile
  • Log / Box-Cox — compress tails
  • Robust scaler — uses median & IQR, not mean & std
  • Huber loss — linear past a threshold
Tolerate them
  • Tree models (XGBoost, LightGBM) — split-based, outlier-robust
  • Quantile regression — predicts medians or percentiles
  • Rank features — replace value with its rank
  • Keep them — if they're the signal, leave them
FLIP
Q: You see |z| = 4.2 on a feature. What do you do?
A: Investigate — is the feature even normal? If skewed, z is misleading; switch to IQR or modified z-score. If genuinely normal and the point is a data error, fix it. If it's real signal (high-value customer, fraud, anomaly), keep it. Auto-dropping is a junior move.

FLIP
Q: When are outliers actually the target?
A: Anomaly detection problems — fraud, intrusion detection, manufacturing defects, predictive maintenance, rare disease. Removing outliers there destroys the signal. Use isolation forest, autoencoders, or one-class SVM.

Class imbalance — the canonical answer

There's a wrong answer here: "I'd just SMOTE it." There's a right answer: a layered approach where SMOTE is the last resort.

The ordered playbook (memorize this order)

  1. Check if it reflects reality. Sometimes the imbalance IS the world (fraud, churn). Don't fight it — measure correctly.
  2. Use the right metric. PR-AUC, per-class F1, recall on the minority class. Drop accuracy.
  3. Stratified splits. Random splits can leave a fold with zero positives — train AND val.
  4. Class weights in the loss. class_weight='balanced' in sklearn, scale_pos_weight in XGBoost. Free, principled, no synthetic data.
  5. Threshold tuning. Don't accept the default 0.5 — pick the threshold from the PR curve that matches business cost.
  6. Resampling, only if 1–5 didn't get there.
    • Random undersampling majority — fast, but discards data.
    • Random oversampling minority — risk of overfitting duplicates.
    • SMOTE — synthetic minority via interpolation. Risky in high dimensions (synthetic points off the data manifold).
    • SMOTE variants — Borderline-SMOTE, ADASYN focus on hard examples.
  7. Anomaly detection framing. If positives are < 1%, treat it as anomaly detection — Isolation Forest, autoencoder, one-class SVM.
"Order of operations: stratified splits, class weights, threshold tuning, then resampling if needed. SMOTE last — it can hurt in high-dim spaces by creating synthetic points off the data manifold."

The SMOTE trap he might set

If he asks "what if I SMOTE first thing?" — push back. Reasons to be cautious:

  • Apply to train only, never val/test. SMOTE-ing the test set inflates metrics — synthetic points are easy to predict.
  • Apply inside the CV fold, not before. Otherwise you leak across folds.
  • Calibration breaks. Predicted probabilities no longer reflect real class priors. Recalibrate (Platt scaling, isotonic regression) afterward.
  • High-dim curse. In high-dim spaces, interpolations between minority points often land in low-density regions where the decision boundary is unreliable.

Threshold tuning — the underrated lever

Most models output probabilities, then threshold at 0.5. That's a default, not an answer. The right threshold comes from business cost:

Cost-sensitive threshold Choose t* that minimizes:  cost = C_FN · FN(t) + C_FP · FP(t)

If a missed fraud costs $1000 and a false flag costs $5 of analyst time, your threshold should be far below 0.5.

FLIP
Q: Why don't you SMOTE the test set?
A: Because the test set must reflect production reality. SMOTE'ing it creates synthetic minority points that are easy to classify (they sit between real minority neighbors), inflating metrics. You'd ship a model that looks great offline and fails in prod. SMOTE the train fold only.

FLIP
Q: 0.1% positive class — your move?
A: Reframe as anomaly detection rather than supervised classification. Try Isolation Forest, one-class SVM, or autoencoder reconstruction error. If supervised, use stratified k-fold, heavy class weights, focal loss for neural nets, and evaluate on PR-AUC + recall@high-precision.

Train / val / test & leakage hunting

Most "amazing offline performance" stories are leakage in disguise. He'll respect rigor here more than any modeling trick.

Choosing the split strategy

Data has...UseWhy
No structureRandom stratifiedPreserves class distribution per fold.
Time componentTime-based split / walk-forward CVRandom splits leak the future into the past.
Grouped entities (users, dealers, docs)GroupKFoldSame entity in train AND test = the model memorizes the entity.
Severe imbalanceStratified k-foldOtherwise a fold can have zero minority samples.
Tiny datasetLeave-one-out or repeated CVMaximize signal, accept high variance.

Leakage — the seven deadly sources

  1. Target leakage: a feature is computed from the target itself or its proxy (e.g. "total_revenue" predicting "made_purchase").
  2. Future leakage: a feature uses information from after the prediction time.
  3. Preprocessing leakage: fitting a scaler / imputer / PCA / SMOTE on the full dataset before splitting.
  4. Group leakage: same user / dealer / document in train and test.
  5. Duplicate leakage: near-duplicates (same content, different IDs) split across train / test.
  6. Temporal contamination: features include aggregations over windows that cross the split boundary.
  7. Hyperparameter leakage: tuning on the test set instead of a separate validation set.
"My first instinct on suspiciously good results is leakage. I check feature provenance, when each feature was computed, and whether any preprocessing fit on the full data before splitting."

Pipeline hygiene that prevents leakage

  • Use sklearn.pipeline.Pipeline — every transform fits on train fold only inside CV.
  • Lock the test set away. Touch it once at the end. No tuning on test.
  • Time-based: TimeSeriesSplit with gap parameter to avoid boundary leakage.
  • For target encoding of high-cardinality features: do it inside CV folds with smoothing, never on full data.
  • Always check feature importance for "too good to be true" features — they're often leaks.

K-fold cross validation — variants you should know

K-foldStandard. K=5 or 10 is the conventional choice.
Stratified K-foldPreserves class ratio per fold. Default for classification.
GroupKFoldEnsures groups don't span folds. Use for per-user, per-doc tasks.
StratifiedGroupKFoldBoth at once. Imbalanced + grouped.
TimeSeriesSplitWalk-forward. Train on past, validate on future. Optionally with gap.
Repeated K-foldK-fold run multiple times with different shuffles. Tightens variance estimates.
Nested CVInner loop tunes hyperparams, outer loop estimates generalization. Honest but expensive.
FLIP
Q: You scaled features then split. What's wrong?
A: Statistics from the test set leaked into training. The mean and std used to scale the train data were computed using test points. Always fit the scaler on train only, then transform train and test separately. Use Pipeline to make this automatic inside CV.
click to reveal
FLIP
Q: Random split vs time-based for predicting next-month churn?
A: Time-based, no question. Random splits would let the model see customers' future behavior in training, which leaks information that won't exist at prediction time. Train on months 1–10, validate on 11, test on 12. Or walk-forward with TimeSeriesSplit.

Feature engineering — the data-centric superpower

This is the section he'll spend the most time on. The JD literally says "Proven feature engineering and data-centric ML skills."

The framework — walk through this on demand

  1. Domain features. What would a human expert look at? Talk to PMs, ops, support — they know the signals.
  2. Aggregations. Counts, means, ratios, sums over windows (last 7d, last 30d, lifetime).
  3. Interactions. Cross features (price × category), polynomial features (rare, only if it helps).
  4. Temporal. Recency (days since last X), frequency, trend (slope of last N), seasonality (day of week, month).
  5. Text. TF-IDF first (cheap, strong baseline), then embeddings if you need semantics.
  6. Categorical encoding. One-hot for low cardinality (<~10), target encoding (with smoothing, inside CV) or embeddings for high cardinality.
  7. Missing values. Don't blindly impute — missingness is often signal. Add an "is_missing" indicator alongside imputation.

Categorical encoding — pick by cardinality

CardinalityMethodNotes
Binary (yes/no)Label (0/1)Trivial.
Low (< ~10)One-hotStandard, no ordering assumed.
Ordinal (small/med/large)Ordinal encodingPreserves order.
Medium (10–50)One-hot OK or target encodingOne-hot getting wide.
High (50+)Target encoding (smoothed) / hashing / embeddingsOne-hot explodes feature space.
Tree modelsNative categorical (LightGBM, CatBoost)No encoding needed if model supports it.

Target encoding — the leakage minefield

Replace a category with the mean target value for that category. Powerful, but:

  • Compute inside the CV fold — never on full data, or you leak.
  • Smooth toward global mean for low-frequency categories: (n·mean_cat + α·mean_global) / (n + α).
  • Use leave-one-out or out-of-fold target encoding for the training fold.
  • For test set, use means computed on the entire training fold.

Scaling & normalization — when each matters

MethodFormulaWhen
StandardScaler(x − μ) / σLinear models, SVMs, neural nets, PCA, KNN. Default for ~normal features.
MinMax(x − min) / (max − min)Fixed range needed (image pixels, NN inputs). Sensitive to outliers.
RobustScaler(x − median) / IQROutliers present.
Log / Box-Cox / Yeo-Johnsoncompress tailsHeavy right tail (income, revenue, durations).
Tree modelsnone neededSplits are scale-invariant.

Missing values — the smart playbook

  • MCAR (Missing Completely At Random) — drop or simple impute, no bias.
  • MAR (Missing At Random, given other features) — model-based imputation (KNN, iterative).
  • MNAR (Missing Not At Random — missingness depends on the missing value itself) — often the signal. Add an indicator. E.g. "income not provided" correlates with low income.
  • Tree models (XGBoost, LightGBM) handle missing natively — let them.
  • Never impute with the mean blindly on skewed features — use median or model-based.

Always have a stupid baseline

Before any fancy feature, compare to:

  • Predict the majority class (classification)
  • Predict the mean / median (regression)
  • Logistic regression on raw features
  • For text: TF-IDF + Logistic Regression

If your fancy model only beats baseline by 1%, you don't have a model — you have a placebo.

FLIP
Q: A categorical feature has 10,000 unique values. What do you do?
A: Don't one-hot. Options: (1) target encoding with smoothing, computed inside CV folds; (2) hashing trick for fast bucketing; (3) entity embeddings learned during training; (4) frequency encoding as a baseline; (5) group rare categories into "other." For tree models, use native categorical support if available (LightGBM, CatBoost).

FLIP
Q: 30% of values are missing in a key feature. What do you do?
A: First, ask why. MCAR, MAR, or MNAR? Add an "is_missing" indicator regardless — missingness is often signal. Then decide imputation: median for skewed numerics, mode for categoricals, or model-based (KNN, iterative imputer). Or use a tree model that handles missing natively. Never just drop rows — you'd lose 30% of your data.

Bias, variance & overfitting

Classic territory. Get the diagnostic right and you sound like someone who's debugged real models.

The decomposition

Expected error Error = Bias² + Variance + Irreducible noise
  • Bias = error from wrong assumptions. Underfitting. Model too simple.
  • Variance = error from sensitivity to training set. Overfitting. Model too flexible.
  • Irreducible noise = inherent randomness; the floor you can't beat.

Diagnose from learning curves

SymptomDiagnosisFix
High train error, high val errorHigh bias — underfittingMore features, more complex model, less regularization, longer training.
Low train error, high val errorHigh variance — overfittingMore data, regularization, simpler model, dropout, early stopping, ensembling.
Both errors plateau, gap closes with more dataHealthyProbably more data still helps; stop or ship.
Both errors plateau, gap stays largeVariance dominantRegularize harder or simplify model.

Regularization — the levers

TypeEffectUse
L1 (Lasso)Pushes weights to zero — feature selectionMany features, suspect most are noise.
L2 (Ridge)Shrinks weights smoothlyDefault. Multicollinearity present.
ElasticNetL1 + L2 combinedBoth effects wanted.
DropoutRandom neuron deactivationNeural nets — implicit ensembling.
Early stoppingStop when val loss plateausUniversal cheap regularizer.
Tree-specificmax_depth, min_samples_leaf, subsample, colsampleXGBoost / LightGBM — your real overfit defense.
Data augmentationMore effective "data"Vision, NLP, audio.

The overfitting checklist when his answer is "I don't believe these results"

  • Did you check val performance, not just train?
  • Did you tune on val, not test?
  • Is there leakage? (See Section 5.)
  • Is the test set representative of production?
  • Is the dataset large enough that the score is stable across seeds / folds?
  • Have you compared to a stupid baseline?
FLIP
Q: Train accuracy 99%, val accuracy 70%. Diagnose and fix.
A: Classic overfitting — high variance. Fixes in order of cheapness: more data (augmentation if possible), stronger regularization (L2, dropout, lower max_depth), simpler model, early stopping, ensembling. Also verify it's not leakage masquerading as overfitting — check splits and feature provenance.
click to reveal
FLIP
Q: Train accuracy 65%, val accuracy 64%. Diagnose.
A: Underfitting — high bias. Model is too simple or features too weak. Fixes: more features (or better ones), more model capacity, lower regularization, train longer. Don't add data — that helps variance, not bias.

Model selection — when to use what

Don't reach for the fanciest tool. The signal of seniority is matching model complexity to the problem.

Quick decision matrix

Problem shapeDefault modelWhy
Tabular, <1M rowsGradient boosting (XGBoost/LightGBM/CatBoost)Mixed types, missing handled, nonlinear, beats deep learning on tabular.
Tabular, interpretability requiredLogistic / linear regression, GAMsCoefficients = explanations.
Text classification (no LLM budget)TF-IDF + Logistic / Linear SVMStrong baseline, fast, interpretable.
Text classification (with LLMs)Few-shot LLM, then fine-tuned encoderFew-shot fast; fine-tune at scale for cost.
ImagesPretrained CNN / ViT + fine-tuneDon't train from scratch unless you must.
Sequences / time seriesGradient boosting on lag features, then LSTM/TransformerBoosting on engineered lags often wins.
RecommendationsMatrix factorization → two-tower → LLM-augStart simple, escalate if needed.
Anomaly detectionIsolation Forest / autoencoderUnsupervised; works with few labels.
Few labels (< 100)Few-shot LLM, weak supervisionDon't fine-tune; you don't have the data.
"For tabular problems I default to gradient boosting before reaching for deep learning. It handles mixed types and missing values natively, trains in minutes, and beats neural nets on tabular up to surprisingly large dataset sizes."

Why XGBoost / LightGBM crush deep learning on tabular

  • Handle mixed data types (numeric + categorical) natively
  • Robust to outliers (split-based, not distance-based)
  • Handle missing values without imputation
  • No need for feature scaling
  • Capture nonlinear interactions automatically
  • Train in minutes on a CPU
  • Mature interpretability via SHAP and feature importance

Deep learning wins on raw signal data — images, audio, long text — where representation learning matters and tabular feature engineering is impossible.

LLM vs classical ML — the explicit framing

LLM wins
  • Low / no labeled data
  • Fuzzy semantics, natural language in/out
  • Long tail of edge cases
  • Fast prototyping window
  • Tasks needing world knowledge
  • Multi-step reasoning over text
Classical wins
  • High volume + low latency
  • Strong tabular features
  • Need calibrated probabilities
  • Cost-sensitive at scale
  • Regulatory / interpretability constraints
  • Deterministic decisions

Hybrid often best: LLM extracts structured features → classical model decides. Gets you LLM flexibility with classical determinism and cost.

FLIP
Q: Tabular dataset, 50k rows, 30 features, binary classification. Default model?
A: Gradient boosting (XGBoost / LightGBM / CatBoost). Beats neural nets on tabular at this scale, handles mixed types and missing values natively, trains in seconds. I'd benchmark a logistic regression baseline first to make sure boosting actually adds value.

FLIP
Q: When would you NOT use an LLM for a classification task?
A: When you have abundant labeled data, need sub-100ms latency at scale, need calibrated probabilities, or face strict cost / interpretability constraints. A fine-tuned encoder or even logistic regression on TF-IDF is often cheaper, faster, and good enough. Reserve LLMs for tasks where their flexibility actually pays for itself.

Error analysis — the data-centric AI move

Andrew Ng's data-centric AI gospel. Aggregate metrics lie. Slicing reveals truth. This is exactly the muscle the JD wants.

The error analysis loop

  1. Pull all wrong predictions from your validation set.
  2. Slice by segment — class, input length, source, time period, entity type, language, etc.
  3. Compute per-slice metrics — find slices where the model is much worse than average.
  4. Read the actual examples. Yes, with your eyes. Patterns emerge in 50 examples that no metric reveals.
  5. Categorize errors — labeling errors, ambiguous cases, missing context, model weakness, distribution shift.
  6. Quantify the categories — what % of errors fall into each? Fix the largest first.
  7. Often the fix is in the data: relabel, augment a slice, add a feature, fix a bug. Not always a new model.
"Aggregate metrics lie — slicing reveals truth. I read errors by segment, categorize them, and quantify each category. Most of the time the fix is in the data, not the model architecture."

Useful slices to compute

  • Per-class — which classes are you worst on?
  • Input length — short inputs vs long; many text models choke on extremes.
  • Source / segment — different dealers, different doc sources, different user cohorts.
  • Time — recent vs old; degradation over time is drift.
  • Confidence — high-confidence errors are scarier than low-confidence ones; they suggest systematic mistakes.
  • Rare patterns — unusual entities, languages, formats.

Confusion matrix reading — fast pattern recognition

  • Diagonal heavy — model works overall.
  • One off-diagonal block bright — two classes are getting confused. Often labeling ambiguity or genuinely similar inputs. Fix: add discriminating features, refine labels, merge classes if they're not distinct in practice.
  • Whole row dim — model never predicts that class. Imbalance issue or threshold problem.
  • Whole column dim — model under-predicts that class. Threshold issue or class weight needed.

The "interesting failures" file

Senior ICs keep one. Every time the model breaks in a surprising way, log the example. Before any architectural change, run the new model against this file. If it doesn't fix the cases that motivated the change, the change isn't doing what you think it's doing.

FLIP
Q: Aggregate F1 is 0.85 but the product team says quality is bad. What do you check?
A: Slicing. Aggregate F1 hides per-slice failure. Compute F1 per class, per source, per input length, per user segment. The 15% errors are probably concentrated in a slice the product team interacts with most. Fix that slice — through data augmentation, relabeling, or a slice-specific feature — before any model change.

FLIP
Q: Most data scientists skip reading actual errors. Why does it matter?
A: Metrics tell you how much wrong, not why. Reading 50 misclassified examples reveals patterns no metric captures — labeling inconsistencies, ambiguous cases, missing context, a class of inputs the model never saw. Often the fix is data work that takes hours and beats a week of model tuning.

LLM evaluation — beyond "looks good to me"

Retails wants someone who can validate LLM-prototypes rigorously. The eval discipline you describe here separates you from people who only "ship demos."

The layered eval framework

  1. Golden set — a fixed, versioned set of ~100–500 hand-labeled examples covering main use cases plus known edge cases. The single most important artifact for trustworthy iteration.
  2. Task-specific automated metrics — exact match, F1 for classification; ROUGE/BLEU for summarization (with caveats); structured output validation (JSON schema match) for tool-calling.
  3. LLM-as-judge — use a stronger model to rate outputs on rubric criteria (faithfulness, fluency, helpfulness). Validate the judge on a sample with humans first.
  4. Human spot-check — sample N production outputs weekly for human review. Catches issues automated eval misses.
  5. Regression set — every fixed bug becomes a test case. Run on every prompt change.
  6. Production monitoring — sample outputs, track latency, cost, refusal rate, user feedback.

LLM-as-judge — the right way

  • Use a stronger model than the one being judged. Otherwise you're checking the model against itself.
  • Validate the judge. Have humans label a sample, check judge agreement (Cohen's kappa). Below ~0.7 → judge is unreliable.
  • Use rubrics, not vibes. Score on specific criteria — factually correct, follows format, stays on topic — not "is it good?"
  • Pairwise > absolute. Asking which of A or B is better is more reliable than rating either alone.
  • Position bias is real. Judges prefer the first or last option — randomize order.
  • Verbosity bias. Judges prefer longer answers — control for length.

Hallucination — how you actually mitigate it

  • Ground in retrieved context (RAG) — the model cites, you check citations.
  • Structured output — force JSON schema; reject malformed.
  • Constrained decoding — only allow tokens from a vocabulary or grammar.
  • Verification pass — second LLM call to fact-check the first against context.
  • Refuse when unsure — system prompt gives the model an "I don't know" path.
  • Tool calls for facts — don't ask the LLM for prices or inventory; have it call your DB.

Force structured output, validate against schemas, and add a verification pass that cross-checks claims against the retrieved context.

Production debugging — "it worked in dev"

Nice-to-have on the JD includes Docker / FastAPI for local validation. He'll respect awareness even if you won't own MLOps.

The five usual suspects

  1. Train/test leakage that wasn't caught. Model looked great offline because it had access to information it doesn't have in prod. Most common cause of "it worked in dev."
  2. Distribution shift. Prod data differs from training. Could be covariate shift (input distribution moves), label shift (class priors move), or concept shift (P(y|x) changes).
  3. Eval set wasn't representative. Too clean, missing edge cases, sampled from a single time slice.
  4. Latency / cost constraints not modeled in dev. Dev used a 7B model, prod budget allowed only 3B.
  5. Upstream data pipeline change. A feature silently changed format or meaning. Schema and contract tests prevent this.
"My five suspects for prod regression are leakage, distribution shift, unrepresentative eval, missed latency/cost constraints, and upstream pipeline drift. I'd add input-distribution monitoring — by the time accuracy drops, you're already late."

Distribution shift — the three flavors

TypeWhat changesDetect with
Covariate shiftP(x) changes; P(y|x) staysKS test, PSI, drift detectors on input features.

Concept driftP(y|x) itself changesMonitor live performance on labeled subset; relearn periodically.

Monitoring — what to watch in production

  • Input distributions — feature drift via PSI (Population Stability Index) or KS test per feature.
  • Output distribution — sudden shift in predicted class proportions.
  • Confidence histogram — if confidence collapses, something changed.
  • Live performance — wherever you can get labels (delayed feedback, sampled human review).
  • Latency p50, p95, p99 — tail latency kills UX more than averages.
  • Cost per call — for LLMs especially.
  • Refusal / fallback rate — when the model declines or errors.

Calibration — often forgotten, often what's wrong

A model is calibrated if among predictions of "70% probability," 70% are actually positive. Many models output uncalibrated scores — gradient boosting tends to over-confident, neural nets too — and your downstream decisions break.

  • Reliability diagram — plot predicted probability bins against observed frequency.
  • Expected Calibration Error (ECE) — single number for calibration quality.
  • Fixes: Platt scaling (logistic on top of model output) or isotonic regression. Train on a held-out calibration set, never the test set.
FLIP
Q: Model accuracy dropped 5% in production overnight. First three things you check?
A: (1) Did the upstream data pipeline change? Schema or feature definition shifts cause silent breaks. (2) Has input distribution drifted? Run PSI or KS against training distribution. (3) Did anything change in the serving stack — model version, preprocessing, library version? Bisect from the deployment. Don't assume model retraining is the answer until you've ruled out infrastructure causes.

Trick questions — opening lines drilled

If nothing else sticks, lock these. The opening line buys thinking time and signals the right framing. The rest will follow naturally.

Q1."Your model has 95% accuracy. Should you ship it?"
"Accuracy alone tells me nothing — I need the baseline and the class distribution before I can answer."
Q2."How would you handle severe class imbalance?"
"First I check whether the imbalance reflects reality — sometimes the right answer is to leave it alone and tune the threshold."
Q3."Build me a classifier from scratch for [problem]."
"Before any modeling, I'd nail down the success metric tied to a business outcome, then build a labeled set, even small."
Q4."It worked in dev but breaks in prod. What happened?"
"Five usual suspects in order — leakage, distribution shift, unrepresentative eval, latency or cost constraints, and upstream pipeline change."
Q5."When would you NOT use an LLM?"
"Whenever a simpler approach gets you 90% of the way there."
Q6."How do you prevent overfitting?"
"Most overfitting I've debugged wasn't a model problem — it was a validation setup problem."
Q7."How do you evaluate an LLM output?"
"Depends on the task — I never use a single metric. I layer task-specific automated metrics, LLM-as-judge, and human spot-check on a versioned golden set."
Q8."You have 100 labeled examples. What do you do?"
"I don't reach for deep learning."
Q9."Train/val/test — how do you split?"
"Default is stratified for classification, but I always check for leakage vectors first — temporal, group, and duplicate."
Q10."How do you do error analysis?"
"Aggregate metrics lie — slicing reveals truth."
Q11."Why does gradient boosting beat deep learning on tabular data?"
"Tabular features are already dense and meaningful — there's no representation learning win — and boosting handles mixed types, missing values, and outliers natively."
Q12."Define a z-score and tell me when not to use it."
"z = (x − μ) / σ — number of standard deviations from the mean. I avoid it on skewed or outlier-heavy data; I switch to IQR or modified z-score with median and MAD."
Q13."Walk me through detecting outliers in a new dataset."
"First I check the distribution shape — that decides the method. Normal-ish: z-score. Skewed: IQR or modified z-score. Multivariate: isolation forest. Then I investigate before removing — outliers can be the signal."
Q14."How do you decide between fine-tuning and prompting?"
"Cost, latency, and data volume. Prompting wins for prototypes and low volume; fine-tuning wins when per-call cost or latency matters at scale and you have a few thousand labeled examples."
Q15."What does a p-value of 0.04 mean?"
"It's the probability of observing data this extreme if the null hypothesis were true — it is not the probability the null is true. I always pair it with effect size."

Meta-tactics for the technical block

  • Don't know? Reason out loud. "I haven't done that specifically. My instinct would be X because Y. Is that the direction you'd take?" — turns it into a conversation.
  • Buy time. Repeat the question in your own words, name the sub-problem, then answer.
  • Don't over-answer. One sentence, then offer to expand. Senior people leave silence.
  • Hook to a project. "...which is exactly what I dealt with on the synthetic data work." Concrete > theoretical.
  • Speak slowly. Pace = confidence. Rushing reads as junior.
