What This App Really Is

This is not a large production platform with many backend services. It is a single local Python application that reads one Excel file, computes retail KPIs and comparisons, and then shows the result in a browser using Streamlit.

The short version: the app takes weekly store data, converts it into a clean analytics payload, raises deterministic flags, and only then asks an LLM to write a short narrative. The LLM is the wording layer, not the analytics engine.
Input Weekly Excel file with raw store columns.
Middle Python logic computes KPIs, YoY, benchmarks, anomalies, and tags.
Output KPI cards, trend chart, transparency panel, and optional narrative.

What Streamlit Is

Streamlit is a Python framework for turning a Python script into a web app quickly. If you know Jupyter, think of Streamlit as a way to make a notebook-like Python workflow interactive in a browser, with dropdowns, charts, buttons, and panels.

When you run streamlit run app/streamlit_app.py, Streamlit starts a local web server. Your browser connects to that local server. The same Python script is rerun whenever the user changes the selected store, selected week, or clicks a button.

Important architecture detail: there is no separate backend API service here. The UI and the analytics logic are both being orchestrated directly by the Streamlit app process.
Question Answer in this project
Do we have a frontend? Yes. The browser page built by Streamlit is the frontend.
Do we have a backend? Yes, but lightweight. The same Python process acts as the backend logic layer.
Do we have a database? No. The source is a local Excel file.
Do we have microservices? No. This is a compact prototype, not a distributed system.

End-to-End Pipeline

The real runtime path is below. This is the kitchen behind the app.

Excel workbook
  -> load_weekly_kpi()
  -> validate()
  -> compute_kpi_tree()
  -> compute_network_reference()
  -> compute_store_vs_network()
  -> compute_yoy()
  -> flag_current_week() + flag_ly_baseline()
  -> build_payload()
  -> generate_tags()
  -> narrate() using fallback or LLM
  -> Streamlit renders cards, chart, narrative, and transparency panel

1. Load the raw file

load_weekly_kpi() in src/retails_insight/loader.py reads the synthetically generated sample workbook with pandas.read_excel() and keeps only the expected columns.

2. Validate obvious data issues

validate() checks for duplicate store-week rows, missing values, and the key retail sanity rule gross_transactions > traffic. That last one matters because it implies conversion rate above 100%.

3. Build the KPI tree

compute_kpi_tree() derives the commercial metrics used in the UI and narratives.

4. Build comparison layers

compute_yoy() creates year-over-year values for the same store and same week. compute_network_reference() computes the weekly network medians and MAD values. compute_store_vs_network() computes the store minus network gaps.

5. Raise anomaly and caveat flags

The anomaly module checks whether the current week looks unusual against store history, and whether last year’s same week was itself abnormal.

6. Convert everything into one typed payload

build_payload() packages KPIs, YoY numbers, network stats, gaps, flags, and DQ caveats into one clean object called StoreWeekPayload.

7. Generate narrative tags

generate_tags() decides which story points are important enough to mention, such as strong sales decline, DQ caveat, suspicious last-year baseline, or underperformance vs network.

8. Write the narrative

If an API key exists, the app can call the LLM. If not, it produces a deterministic fallback summary. Either way, the numbers and flags come from the same payload.

CR, UPT, AUP and the KPI Tree

The app starts from four raw business measures: traffic, gross_transactions, gross_quantity, and net_sales. Everything else is derived from those four.

CR Conversion Rate = gross_transactions / traffic
Meaning: of the visitors who came in, how many converted into transactions.
UPT Units Per Transaction = gross_quantity / gross_transactions
Meaning: basket size, or how many units were sold per purchase.
AUP In the code this is avg_selling_price = net_sales / gross_quantity
Meaning: average price per unit sold.
ATV Average Transaction Value = net_sales / gross_transactions
Meaning: average value of each purchase.

net_sales = traffic × conversion_rate × units_per_txn × avg_selling_price

This identity is central to the app. It means the sales story can be broken into four driver questions:

  1. Did fewer shoppers come in?
  2. Did fewer shoppers convert?
  3. Did baskets get smaller?
  4. Did selling price per unit go down?
Why this matters: instead of saying “sales are down” in a vague way, the app can say which commercial lever most likely drove the decline.

What the EDA Changed

The EDA was not decoration. It directly influenced the design. The findings are summarized in docs/eda_summary.md and the full notebook is in notebooks/EDA.ipynb.

EDA finding Why it mattered What design choice came from it
The retail KPI identity held to floating-point precision. It proved the derived KPIs were trustworthy. The app computes CR, UPT, AUP and decomposition deterministically instead of storing them manually.
2025 stops at week 48. Naive YoY across non-overlapping weeks would be misleading. YoY logic compares same store and same week only.
Store_G 2025 W21 had gross_transactions > traffic. This creates an impossible conversion rate over 100%. The loader raises an explicit DQ caveat rather than hiding it.
The distributions were skewed and included outliers. Mean and standard deviation can become unstable. The anomaly logic uses median and MAD instead of standard z-scores.
Some LY baselines were themselves abnormal. YoY can be mathematically correct but commercially misleading. The app flags abnormal LY baselines and warns that YoY may be unreliable.

Why Pydantic Was Used

Pydantic is a Python library for defining strict data models. In this app, it is used in payload.py for StoreWeekPayload and NarrativeResult.

Reason 1: contract enforcement It guarantees the payload has the fields the narrative layer expects.
Reason 2: early failure If a field is missing or an unexpected field appears, the app can fail loudly instead of silently producing bad output.
Reason 3: JSON-ready objects The payload can be converted cleanly into JSON for the LLM prompt and transparency panel.
Reason 4: testability The eval harness can validate synthetic scenarios directly against the same schema as runtime data.

One especially important choice is ConfigDict(extra="forbid"). That means extra unexpected fields are rejected. This prevents accidental payload drift.

How Tags Are Raised

The app does not ask the LLM to decide what is important. It decides that itself using deterministic tag rules in src/retails_insight/tags.py.

Tag family Rule Purpose
sales_yoy_strong_decline net_sales_yoy <= -10% Flags a meaningful commercial drop.
*_drove_decline Driver attribution share is at least 50% Names the main cause of the decline.
ly_baseline_suspect_* Any ly_baseline_abnormal_* flag exists Warns that YoY may be misleading.
dq_caveat_* Any DQ issue exists for the selected store-week Forces narrative transparency on bad input data.
network_underperform_* Store gap is negative and materially below network, usually beyond weekly MAD Shows relative underperformance, not just absolute change.
What this means in plain English: the LLM is not deciding whether the store had a strong decline or a DQ caveat. The app already made that decision before the model wrote a sentence.

How the LLM Is Controlled

The LLM integration is intentionally constrained.

  1. The app first builds a structured payload.
  2. It also builds a structured tag list.
  3. Both are serialized into JSON and put into the prompt.
  4. The model is told Do not invent numbers.
  5. The response is forced into a JSON schema with required fields.
  6. If the LLM fails, the app falls back to deterministic text.

The schema requires fields such as summary, flags_narrated, yoy_caveat_present, network_gap_mentioned, and dominant_driver_cited. This is important because it makes the output machine-checkable.

Operational detail: the app only calls the LLM when the user clicks Generate narrative. Browsing store/week combinations alone does not trigger an API call. This controls cost and latency.

Why the YAML File Exists

eval/golden_set.yaml is not part of the runtime user flow. It is part of the evaluation system.

The YAML file contains a list of synthetic store-week scenarios. Each scenario has an id, a description, and a payload already shaped like StoreWeekPayload.

Why YAML instead of Python code? YAML is easy to read, easy to diff in git, and easy for a PM or reviewer to inspect.
Why not use the real Excel for eval? Because eval needs stable, reproducible scenarios, including edge cases that may be rare in the real file.
Why not JSON? YAML is more human-friendly for manually authored scenario libraries.

In other words, the YAML file is the test case catalog for the narrative system.

How the Evaluation Harness Works

The evaluation harness lives mainly in src/retails_insight/judge.py and is run through python -m retails_insight.eval.

What happens during eval

  1. Load the YAML scenarios.
  2. Validate each payload with the same Pydantic schema used at runtime.
  3. Generate tags for that payload.
  4. Generate a narrative, using the LLM if available, or fallback text if not.
  5. Run a set of checks against the result.
  6. Write a markdown report to eval/reports/eval_v1.md.

What the checks actually test

Check Meaning
numeric_grounding If the narrative mentions a number, that number must already exist in the payload, allowing for percentage rendering.
tag_coverage If payload flags exist, the narrative must say it addressed them.
yoy_caveat_check If ly_baseline_abnormal is true, the narrative must warn that YoY is unreliable.
network_gap_check If a flagged KPI is below network, the narrative must mention that gap.
no_hallucinated_flags The narrative cannot claim decline, anomaly, DQ or underperformance if the deterministic tags did not support that claim.
llm_judge Optional second-model review if a separate judge model is configured.
This is the main safety mechanism: the app does not just hope the LLM behaves. It checks whether the narrative is grounded in the payload and whether required caveats were present.

What Is Hard-Coded

Yes, there are hard-coded choices. That is normal in a feasibility prototype. The important question is whether they are explicit and justified. In this codebase, most of them are explicit.

Hard-coded item Where Current value

PM framing: these are product and analytics policy choices. In a production system, many would likely move to config files or an admin layer. For a prototype, keeping them in code is reasonable because it keeps the behavior explicit and testable.

Why These Design Choices Make Sense

Why Streamlit?

Because it is the fastest way to deliver an interactive demo in pure Python. For an AI PM assignment, the point was to prove the logic, transparency, and evaluation approach, not to spend most of the time building a production web stack.

Why deterministic-first?

Because retail KPI math should not be delegated to an LLM. The deterministic-first approach makes the app auditable. The model only writes language around already-computed facts.

Why medians and MAD instead of averages and standard deviation?

Because EDA showed outliers and skew. Robust statistics are less easily distorted by a single bad week.

Why a payload object between analytics and narrative?

Because it creates a clean contract. The narrative layer gets a stable input format. The UI can show that same payload in the transparency panel. The eval harness can reuse the same structure.

Why a fallback narrative?

Because the app still needs to work if there is no API key, an outage, or a bad model response. That is good product behavior.

Why anonymise store names for the LLM?

Because privacy discipline is easier to defend when raw identifiers do not cross the API boundary unnecessarily.

The most important message to carry into a discussion: this application is designed so that the risky part, the generative language layer, is the smallest and most controlled part of the stack.
